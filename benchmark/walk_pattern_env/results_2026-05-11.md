# Synthetic walk-pattern benchmark — 3-way comparison + raw Tokenizer POC

Run date: 2026-05-11
Julia 1.12.6, Darwin aarch64.

Workload: synthetic XML document with N placemark-like nested elements, each
of depth ≥ 4 (`<root>/<placemark>/<point>/<coordinates>`). Generated by
`walk_pattern.jl::generate_doc(N)`. Walker accumulates `length(value)` over
every Text/CData node, recursively visiting every Element child.

Strategy 5 ("raw Tokenizer DFS") directly uses `XML.Tokenizer` +
`TokenizerState` (imported into `XML` namespace from `.XMLTokenizer` but
**not exported**) without any wrapper (no `Stateful`, no `LazyChildIterator`,
no `LazyNode`). Tests the hypothesis: if v0.4 exposed the tokenizer
publicly, what perf class could we reach?

`@benchmark` budget: 3 seconds per strategy per N.

## Headline observations

1. **PR #58 (ctx-share) gains ≈ -27% time / -25% memory on `LazyNode + children()`** — wider scope than the PR description.
2. **PR #59 (`next!` DFS) is ×3.3 faster and ×7 less memory than the best `children()` path on v0.3+#58**, with 1.9M allocs instead of 10.8M.
3. **v0.4 regresses `LazyNode + children()` vs v0.3+#58**: ×1.9 slower, ×1.5 more memory. The intrinsic `ctx` savings are more than offset by new allocations (`LazyChildIterator` + `Stateful(Tokenizer)` per `eachchildnode` call, even from `children` path internally).
4. **`eachchildnode` in v0.4 does not recover the `next!` gain**: marginally worse than `children()` on this flat profile.
5. **raw Tokenizer DFS (v0.4 private API, **no LazyNode constructed**) does most of the work**: ×5.2 faster than `eachchildnode` and ×4.3 less memory. By directly using `XML.Tokenizer` instead of `eachchildnode`, the v0.4 public-API regression is largely recovered — *for consumers that don't need a LazyNode wrapper per child*.
6. **Strategy 6 (raw Tokenizer + recursive + LazyNode construction per child) is the FastKML-realistic case**: 308.9 ms / 12M allocs / 1062 MiB. Only ~17% faster than `eachchildnode` (370 ms) — the wrapper allocations (`LazyChildIterator`, `Stateful`, `Ref`) are only a minor share of the cost. The bulk of the cost (~64% of `eachchildnode`'s time) is the **`LazyNode` allocation per yielded child** plus recursion overhead.

### Cost decomposition (v0.4 lazy walk on N=100k synth)

Comparing strategies 3, 5, 6 reveals the structure:

| Component                                       | Cost    | % of `eachchildnode` total |
|-------------------------------------------------|---------|----------------------------|
| Tokenization (strategy 5, flat DFS, no LazyNode)| 72 ms   | 19% |
| **LazyNode allocation per child + recursion**    | **237 ms** | **64%** |
| `eachchildnode` wrappers (Stateful + LazyChildIterator + Ref) | 61 ms | 17% |
| **Total (strategy 3 — eachchildnode)**          | 370 ms  | 100% |

7. **Net implication for consumers building typed wrappers** (FastKML pattern):
   - Exposing `Tokenizer` publicly recovers the 17% wrapper share (option β alone).
   - Recovering the 64% per-child LazyNode allocation cost requires either:
     - **α** a callback-style `walk_children(node, callback)` where `@inline` lets the LazyNode never escape the call frame, or
     - **γ** an opt-in `MutableLazyNode` that one wrapper is reused across the walk (v0.4-shape of PR #59), or
     - **δ** a poolable / reusable `LazyChildIterator` (insufficient alone).
   - The residual `iterate(::Tokenizer, ::TokenizerState)` allocation (~1 per token, ~3M on N=100k) comes from the `Union{Nothing, Tuple{Token, TokenizerState}}` return type exceeding Julia's SROA stack-allocation threshold. Separately optimizable (Tuple inlining, sentinel state).

  using Pkg
  Pkg.add(["BenchmarkTools", "Printf"])

  using Pkg; Pkg.add(name="XML", version="0.3.8")

# v0.3.8 + #58 + #59 (local clone of mathieu17g/XML.jl dev-combined)
